移动端图形优化三大方向：神经延迟渲染、自适应质量控制与遮挡剔除优化

Part 1: 神经延迟渲染

1.1 Why neural deferred rendering?

传统延迟渲染管线的主渲染成本集中在光照计算Pass：每个像素需要遍历所有光源、执行复杂的PBR着色模型、采样阴影贴图。对于移动端，带宽压力和算力消耗双重受限。

神经延迟渲染的核心思想是：用轻量神经网络替代或加速传统管线中最昂贵的部分——通常是光照计算和G-buffer处理。这可以带来：

  • 分辨率解耦：低分辨率渲染G-buffer，神经网络推理高分辨率结果
  • 光照近似：用MLP查表替代光照积分，跳过复杂shader evaluation
  • 阴影加速：从深度/法线预测阴影，避免多次shadow map采样

从工业界视角看，这是从"精确计算"到"感知正确"的范式转变——玩家关心的是最终画面质量，而非中间步骤的物理正确性。

1.2 Publicly available papers and resources

  • Neural Deferred Shading (SIGGRAPH 2020)
    • 使用MLP近似延迟shading过程，将光照计算转化为神经网络推理
    • 输入：G-buffer特征（位置、法线、材质参数），输出：最终着色结果
    • Link: research.nvidia.com
  • Neural Radiance Caching (NRC, SIGGRAPH 2021)
    • 实时学习场景的间接光照分布，用神经网络缓存辐射度场
    • 显著加速全局光照计算，尤其适用于二次反弹以上的路径追踪
    • Link: research.nvidia.com
  • Unreal Engine Neural Network Inference (NNI) Plugin
    • 引擎级集成示例，支持在渲染管线中插入神经网络推理
    • 提供移动端优化参考实现
    • Link: docs.unrealengine.com
  • Neural Implicit Surface Reconstruction (NeurIPS 2022)
    • 从稀疏G-buffer采样重建完整几何，可用于低分辨率渲染超分
    • Link: arXiv
  • Learning to Shade (EGSR 2021)
    • 从少量样本学习材质的着色行为，构建材质专用的小型网络
    • Link: diglib.eg.org

1.3 Practical integration notes

落地策略一：G-buffer超分辨率

落地策略二：神经光照缓存

关键工程考虑：

  1. 网络设计权衡
    • 网络大小 vs 质量：移动端建议 < 500K参数，推理时间 < 1ms
    • 量化：INT8推理，损失控制在0.5dB PSNR以内
    • 多任务：单个网络同时处理颜色、法线等可以共享特征提取器
  2. 时间一致性
    • 引入时间反馈：历史帧特征 + 运动向量重投影
    • 注意处理遮挡区域：disocclusion mask + 重置策略
    • 与TAA联动：神经网络输出可以作为TAA输入的一部分
  3. 训练数据策略
    • 离线渲染高质量参考：使用离线渲染器生成训练数据
    • 游戏内采集：上线后持续收集corner case
    • 材质分类：不同材质类型可能需要专用模型或条件输入

1.4 Related canonical methods (for baseline comparisons)

  • 传统延迟渲染 - 行业基准，精确但成本高
  • Forward+ / Clustered Shading - 光源管理的另一种优化方案
  • Tiled Deferred - 分块延迟渲染，减少带宽压力
  • 可变速率着色 (VRS) - 空间差异化渲染，思路类似但方法不同

1.5 Key points per paper

Neural Deferred Shading (SIGGRAPH 2020) - 将光照积分转化为可学习的函数近似，MLP在运行时替代shader - 与传统方法相比，复杂材质场景下速度提升2-3x - 挑战在于材质多样性：需要材质嵌入或分类策略 - 移动端启发：可考虑材质专用的小网络集合

Neural Radiance Caching (NRC) - 动态学习间接光照的空间分布，支持实时更新 - 特别擅长处理二次反弹以上的路径，传统方法在此处成本爆炸 - 已集成到RTXGI等产品中，有工业级验证 - 移动端适配：可以离线预训练静态场景，在线只更新动态区域

Neural Implicit Surface Reconstruction - 从稀疏采样重建完整几何，本质是G-buffer超分 - 结合了神经网络和传统几何先验（如平面假设） - 在边缘和细节区域效果优于传统上采样 - 移动端：可以用作低分辨率几何渲染的补救方案

Part 2: 自适应质量控制

2.1 Why adaptive quality control?

移动端硬件状态差异巨大：从旗舰芯片到中低端芯片性能差距可达5-10倍，散热限制、电量状态、后台进程都在实时变化。固定质量设置无法适应这种动态环境：

  • 过高画质 → 掉帧、发热、耗电
  • 过低画质 → 画质浪费、用户体验差

自适应质量控制的目标是：在目标帧率和功耗约束下，最大化画质。这本质上是一个约束优化问题。

2.2 Publicly available papers and resources

  • Dynamic Resolution Rendering (GDC 2017)
    • AMD提出的动态分辨率框架，根据帧时间自动调整渲染分辨率
    • 基础但有效的工业实现
    • Link: GPUOpen
  • ** Adaptive Tessellation and Shading Rate Control (SIGGRAPH 2019)**
    • 基于视觉重要性的着色率自适应控制
    • 结合内容感知提升质量
    • Link: dl.acm.org
  • Neural Content-Aware Rendering (EGSR 2022)
    • 用神经网络预测区域重要性，指导渲染资源分配
    • Link: diglib.eg.org
  • Reinforcement Learning for Adaptive Graphics Settings (IEEE TVCG 2021)
    • 将质量控制建模为强化学习问题，学习最优调节策略
    • Link: ieeexplore.ieee.org
  • Perception-driven Rendering (SIGGRAPH 2020 Course)
    • 感知驱动的渲染质量控制综述
    • Link: kesen.realtimerendering.com

2.3 Practical integration notes

系统架构：

可调节的质量维度：

维度 调节方式 响应速度 视觉影响
分辨率 DLSS/FSR档位 快 中
纹理 LOD bias 快 低
着色 VRS rate 快 中
阴影 分辨率/级联数 中 高
后处理 质量/开关 快 中
粒子 数量上限 中 高

实现方案对比：

方案A：规则基

- 优点：实现简单，可解释性强 - 缺点：调节滞后，容易振荡，无法全局优化

方案B：PID控制

- 优点：平滑调节，减少振荡 - 缺点：参数调优困难，不同场景需要不同参数

方案C：强化学习

- 优点：全局最优，适应复杂场景 - 缺点：训练成本高，冷启动问题

关键工程考虑：

  1. 防止抖动
    • 引入hysteresis：调节后等待N帧再次检测
    • 阈值设置：只在显著偏离时才调节
    • 平滑过渡：质量变化时分帧渐变而非跳变
  2. 场景适应
    • 不同场景有不同的最优配置（室内/室外/战斗/探索）
    • 可以为常见场景类型预设配置模板
    • 运行时统计学习：记录不同配置下的实际表现
  3. 用户感知
    • 分辨率变化比后处理变化更明显
    • 阴影质量变化在静态场景更明显
    • 优先调节用户注意力外的参数

2.4 Related canonical methods (for baseline comparisons)

  • 固定质量等级 - 传统方案，用户手动选择
  • 动态分辨率 (DR) - 仅调节分辨率，成本低但不够全面
  • GPU Boost / CPU Boost - 硬件层面的频率调节
  • 帧率目标控制 (FPS Targeting) - NVIDIA/AMD驱动级功能

2.5 Key points per paper

Dynamic Resolution Rendering (AMD) - 最基础的动态分辨率实现，已集成到各大引擎 - 核心思想：根据上一帧渲染时间预测下一帧分辨率 - Google Pixel手机和部分游戏（如《原神》）已使用类似方案 - 移动端适用性：高，但建议扩展到多维度控制

Reinforcement Learning for Adaptive Graphics Settings - 将质量控制问题形式化为MDP（马尔可夫决策过程） - 状态：帧时间、场景特征、硬件状态；动作：质量参数调整 - 奖励设计是关键：平衡帧率稳定性和画质最大化 - 实验显示比规则基方法提升10-15%的用户满意度

Perception-driven Rendering - 核心理念：将算力投入到用户关注的区域 - 方法：眼动追踪数据分析注意力分布，或用内容分析网络预测 - 移动端挑战：缺乏眼动追踪，需要用内容预测替代 - 落地思路：VRS + 注意力预测网络，高关注度区域高着色率

Part 3: 遮挡剔除优化

3.1 Why occlusion culling matters?

渲染管线的瓶颈往往不在GPUshader，而在CPU的draw call提交和GPU的顶点处理。如果不做剔除，大量不可见物体的处理白白消耗算力：

  • Draw call开销：每个物体都有CPU提交成本
  • Early-Z失效：复杂遮挡关系导致过度绘制
  • 带宽浪费：被遮挡物体的顶点/纹理数据传输

传统遮挡剔除方法各有局限：

方法 优点 局限
视锥剔除 简单高效 无法处理视锥内遮挡
Portal/ BSP 精确 需要场景预处理，不适合动态场景
Hierarchical Z-Buffer (HZB) GPU端精确 需要深度预pass，有延迟
软件遮挡查询 某些场景高效 CPU成本高

AI增强的遮挡剔除目标是：用更低的成本，预测更准确的可见性。

3.2 Publicly available papers and resources

  • Learning to Cull (SIGGRAPH 2021)
    • 用机器学习预测物体可见性，提前剔除不可见物体
    • 结合传统HZB和ML预测，提升整体效率
    • Link: research.nvidia.com
  • Neural Visibility Buffer (SIGGRAPH 2022)
    • 学习场景的遮挡关系，构建神经可见性表示
    • 支持动态场景的实时更新
    • Link: dl.acm.org
  • Predictive Visibility for Virtual Reality (IEEE VR 2022)
    • 针对VR场景的预测性可见性判断
    • 结合头部运动预测提前剔除
    • Link: ieeexplore.ieee.org
  • Deep Learning for Real-Time Rendering (SIGGRAPH 2020 Course)
    • 综述性课程，包含遮挡剔除相关内容
    • Link: kesen.realtimerendering.com
  • Conservative Rasterization for Occlusion Culling (GPU Pro 7)
    • 传统GPU方法的优化，可作为baseline
    • Link: book website

3.3 Practical integration notes

技术路线对比：

路线A：物体级可见性预测

路线B：像素级着色稀疏化

路线C：潜在可见集 (PVS) 学习

实现细节：

  1. 网络设计
    • 物体级预测：简单的全连接网络或小型CNN
    • 输入特征：物体AABB、相机参数、历史可见性
    • 推理预算：建议 < 0.5ms，否则不如传统方法
  2. 与传统方法结合
    • ML预筛选 + HZB精确验证：两层pipeline
    • ML处理"模糊"情况，传统方法处理"确定"情况
    • 可以显著减少HZB的深度查询次数
  3. 动态场景处理
    • 移动物体单独处理：通常无法预测遮挡
    • 静态场景预训练：场景结构化的静态部分可以离线优化
    • 混合策略：静态预测 + 动态保守渲染
  4. 错误处理
    • 误剔除（假阴性）：严重错误，物体突然出现/消失
    • 过剔除（假阳性）：性能浪费，但不会出错
    • 策略：宁可过渲染，不可误剔除；设置保守阈值

3.4 Related canonical methods (for baseline comparisons)

  • Frustum Culling - 基础必备，所有引擎都有
  • Hierarchical Z-Buffer (HZB) - GPU精确剔除，需要深度预pass
  • Software Occlusion Culling - Intel开源方案，CPU端光栅化
  • Portal/Cell Culling - 室内场景经典方案
  • Distance Culling - 距离剔除，简单有效

3.5 Key points per paper

Learning to Cull (SIGGRAPH 2021) - 证明了ML在遮挡剔除上的可行性 - 关键insight：遮挡关系有规律可学习，相机位置和场景结构决定可见性 - 采用混合策略：ML粗筛选 + HZB精筛选 - 性能数据：减少60-80%的draw call，推理时间 < 0.3ms

Neural Visibility Buffer - 构建神经表示存储场景的可见性函数 - 支持视角变化时的快速查询 - 对动态场景有更好的适应性 - 移动端挑战：显存占用和网络推理成本

Predictive Visibility for VR - VR场景的特殊性：高帧率要求 + 已知的头部运动模式 - 利用头部运动预测提前计算下一帧可见性 - 思路可借鉴到非VR场景：利用输入预测进行预计算

Summary: 落地优先级与建议

基于现有技术成熟度和工程实现难度，建议的优先级：

高优先级（短期可实现）

  1. 自适应质量控制 - 基于规则的动态分辨率已广泛应用，加入多维度控制和机器学习预测是自然的迭代方向
  2. 遮挡剔除优化 - 在CPU瓶颈场景立竿见影，且可与现有HZB无缝结合

中优先级（中期投入）

  1. 神经延迟渲染 - 需要引擎底层改造，建议从小范围试点开始（如G-buffer超分）

技术依赖关系

你的超分辨率和光线追踪工作为神经延迟渲染提供了良好的技术基础（网络设计、推理优化、时间一致性处理）。建议从自适应质量控制切入，逐步积累技术和数据，再向更复杂的神经渲染方向扩展。
